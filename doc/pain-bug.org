* Debugging process

** Problem

~ai-audit last-session~ uses tmux scrollback fingerprinting to detect
which OpenCode session is displayed in a pane. The command was failing
on many panes with "Could not detect last session", preventing session
resurrection after tmux-resurrect restores.

** Evidence collection

1. Identify all tmux panes where ~ai-audit last-session~ fails.
2. For each failing pane, create a folder in =doc/pane-bug/= named
   after the tmux symbolic address (e.g. =0k-charms:1.1=,
   =opencode:1.2=).
3. In each folder, save the raw scrollback:
   : tmux capture-pane -t "<session>:<window>.<pane>" -p -S -5000 > scrollback.dat
4. Run ~ai-audit~ with verbose logging and save the trace:
   : ai-audit -vvv last-session --scrollback-file <folder>/scrollback.dat \
   :   2> <folder>/run-fails.log
5. Run the analysis script to produce a structured summary:
   : python3 doc/pane-bug/extract-filters.py --save

This generates an =analysis.txt= per folder showing: filters built,
sessions searched, partial matches, hit rates, and final result.

** Iterative fix loop

For each iteration:

1. *Pick the next evidence folder* and read its =analysis.txt=.

2. *Diagnose the root cause* — compare the filter criteria against
   what the DB actually stores. Typical patterns:
   - Filter text exists in a part type that isn't searched
   - Filter text is truncated by TUI rendering
   - Filter text contains TUI chrome (prefixes, escapes) not in DB
   - Filter text comes from non-session content (prompt, title)

3. *Document the bug* as a new heading in this file with evidence
   and proposed fix.

4. *Implement the fix*, add a test, run ~cargo test~.

5. *Rebuild*: ~cargo build --release~.

6. *Re-run all remaining folders* with the new binary:
   : for d in <folders...>; do
   :   ai-audit -vvv last-session --scrollback-file "doc/pane-bug/$d/scrollback.dat" \
   :     2> "doc/pane-bug/$d/run-fails.log" && echo "OK" || echo "FAIL"
   : done

7. *Regenerate analysis*:
   : python3 doc/pane-bug/extract-filters.py --save

8. *For each newly resolved folder*:
   a. Check the tmux pane using its symbolic name:
      : tmux capture-pane -t "<name>" -p | tail -15
   b. If the pane has a free prompt and the resurrection command is
      visible in history, send it:
      : tmux send-keys -t "<name>" \
      :   'LAST_SESSION=$(ai-audit last-session) && oc -s "$LAST_SESSION"' Enter
   c. Wait ~15s, verify the OpenCode TUI loaded.
   d. Delete the evidence folder: ~rm -rf "doc/pane-bug/<name>"~

9. *If the folder still fails* — skip it, leave it for the next
   iteration.

10. *Loop* from step 1 with the remaining folders.

* Bug #1: TextContains filter doesn't match text that ~list-sessions -s~ finds

The ~last-session~ fingerprinting builds ~TextContains~ filters from
the TUI scrollback, but substring matching during session search
doesn't find text that ~list-sessions -s~ successfully finds for the
same needle.

Evidence from =0k-charms:1.1=: filter[2] uses
~TextContains("Done. iroh-relay is now running on vps-03.0k.io...")~
(extracted from assistant text at depth=1). The correct session
=ses_30f15c4f2ffeoZwVg95z8WYjkH= was found by this filter. But
filter[0] uses a shell prompt string extracted from a tool output
block — the search reports "3 text parts in pool / no text part
contains the needle", yet ~ai-audit list-sessions -s~ with the same
substring finds the session.

This means the ~TextContains~ matching in the fingerprint search path
does not use the same code path as ~list-sessions -s~. The
~list-sessions~ search presumably checks all part types (text,
tool_result, etc.), while the fingerprint matcher only checks ~text~
parts, missing content stored in ~tool_result~ parts.

*To investigate*: why ~TextContains~ in fingerprint matching skips
~tool_result~ parts, and whether unifying with the ~list-sessions -s~
search path would fix it.

*Status*: FIXED. ~TextContains~ now calls ~part_contains_needle~
(same as ~list-sessions -s~), searching all part types. Resolved
10/25 evidence cases.

* Bug #2: ToolFieldEquals uses exact match on TUI-truncated values

~ToolFieldEquals~ compares the filter value against the stored tool
argument using exact equality (~s == resolved~, ~db.rs:405~). But the
filter value is extracted from the TUI scrollback, which truncates
long tool arguments at the terminal width. The truncated string never
equals the full stored value.

Evidence from =0k-charms:1.3=: filter[2] at depth=6 has criterion
~ToolFieldEquals(tool=interactive_bash, tmux_command="send-keys -t
%581 \"Rathole server connection details for vps-03.0k.io: Server")~.
The DB stores the full value: ~"send-keys -t %581 \"Rathole server
connection details for vps-03.0k.io: Server host: vps-03.0k.io --
Control port: 2333 -- Service name: vigil -- Service token:
4b299d60b440..."~. The session has 230 messages, the tool call is at
position 3 from the end (well within the search window), the part
structure is correct (~type: "tool"~, ~state.input.tmux_command~), but
the exact comparison fails because the filter only has the truncated
prefix.

The trace confirms: the matcher finds the tool, compares the field,
and reports ~NO MATCH~ because of the truncation.

*Fix*: Replace ~s == resolved~ with ~s.starts_with(&resolved)~ (or
~s.contains(&resolved)~) in the ~ToolFieldEquals~ matching arm. The
TUI-extracted value is always a prefix of the real value, so
~starts_with~ is the correct semantic.

*Status*: FIXED. Resolved 1 additional case (0k-charms:1.3).

* Bug #3: Scrollback parser picks up unsent text from TUI input prompt

The OpenCode TUI has an input/compose area where the user types
messages before sending. Text typed but never sent is rendered in the
TUI and captured by ~tmux capture-pane~. The scrollback parser treats
this text as a session message and builds a ~TextContains~ filter from
it. Since the text was never sent, it is never stored in any session's
messages/parts — the filter is guaranteed to never match.

Evidence from =immich-test:1.1=, =monujo:3.4=, =talk-rs:1.7=,
=talk-rs:1.8=, =tmux-resurrect:2.2=: filter[1] is
~TextContains("LAST_SESSION=$(ai-audit last-session) && oc -s
\"$LAST_SESSION\"")~. This is a shell command the user typed into the
OpenCode input prompt (likely by mistake — intended for bash). It was
never sent, so no session contains it. Yet it generates a filter that
wastes a filter slot and causes false-positive partial matches on
unrelated sessions.

*To investigate*: how the scrollback parser identifies message
boundaries — it needs to detect and exclude the TUI input/compose
area.

*Status*: FIXED. Detect footer fg color, skip prompt lines above it
with same color, plus trailing blank + ~▣~ marker. Resolved 3
additional cases.

* Bug #4: Session title line treated as message content

The OpenCode TUI displays the session title in the panel header as a
bold ~#~ heading with token/cost stats at the end:

: # Storage template migration impact                 157,714  16% ($0.00)

The scrollback parser treats this as ~AssistantText~ and builds
~TextContains~ filters from it. This wastes filter slots — the title
is session metadata, not message content. It also produces duplicate
filters when the title appears at multiple scroll positions (e.g.
=immich-test:1.1= has filter[1] and filter[2] both containing
~"Storage template migration impact"~ at different depths).

The title line has a characteristic format: bold ~#~ at the start of
the text, followed by numbers at the end with a distinctive
presentation (token count, percentage, cost in parentheses).

*To investigate*: detect session title lines in ~classify_tui_line~
and exclude them from message collection, or filter them out during
~build_filters~.

*Status*: FIXED. Detect ~SessionTitle~ lines (bold ~#~ + cost stats)
and stop upward collection. Resolved 5 additional cases.

* Bug #5: TextContains needle includes TUI ~$ ~ command prefix

The OpenCode TUI renders tool command invocations with a ~$ ~ prefix
(e.g. ~$ compose run odoo16 ...~). This prefix is TUI chrome — the DB
stores just the command without it (~compose run odoo16 ...~). When
the scrollback parser extracts text from these lines, the ~$ ~ is
included in the ~TextContains~ needle, causing a mismatch.

Evidence from =monujo:1.4=: filter[0] is ~TextContains("$ compose run
odoo16 --stop-after-init ...")~. The session ~ses_30f33320a...~ was
found by filter[1] (an Odoo log line), but filter[0] failed because
the DB stores ~"compose run odoo16 --stop-after-init ..."~ without the
leading ~$ ~.

*Fix*: strip the leading ~$ ~ from text lines when building
~TextContains~ filter criteria.

*Status*: FIXED. Resolved 1 additional case.

* Bug #6: OSC escape sequences not stripped from filter text

The ~strip_ansi~ function only removes CSI sequences (~\x1b[...~) but
not OSC sequences (~\x1b]...\x1b\\~). OSC 8 hyperlink escapes
(~\x1b]8;id=...;URL\x1b\\~) are used by the OpenCode TUI for
clickable links. When ~longest_ansi_segment~ splits text by CSI
escapes, it treats OSC escapes as regular text, producing segments
that contain raw escape codes. These become ~TextContains~ needles
that can never match the DB (which stores plain text).

Evidence from =opencode:1.2=: only 1 filter was built, containing
~\u{1b}]8;id=589887;https://github.com/...~ — an OSC hyperlink. Zero
sessions matched.

*Fix*: treat OSC sequences as segment breaks in ~ansi_segments~ /
~strip_ansi~, the same way CSI sequences are handled. Any OSC
sequence should split the text so it never ends up inside a filter
needle.

     What the Storage Template does
     It controls how uploaded assets (via mobile app, web UI, CLI) are organized on disk. When enabled, it moves
     those files from UPLOAD_LOCATION/upload/<userID> to UPLOAD_LOCATION/library/<userID> following your chosen

     What the Storage Template Migration job does
     It re-organizes already uploaded assets to match a new or changed template. For example, if you change from Year/
     Month/Filename to Year/Album/Filename, running the migration job moves the existing uploaded files accordingly.

     So you can safely enable the storage template without worrying about your external library being reorganized or

     Architecture: 3-Layer Defense

     Layer 1 — Prevention at Ingestion (Primary fix)

     New file: src/util/image.ts

     A centralized image utility that:
     1. Reads dimensions from raw bytes (PNG IHDR chunk, JPEG SOF marker, GIF header, WebP VP8 header — all can be
     parsed from the first few hundred bytes without a library)
     2. If any dimension > a configurable max (default 8000), resizes using sharp (already a common Bun-compatible
     dependency) or — if you prefer zero dependencies — shells out to ImageMagick convert (widely installed, and a
     missing binary just means "skip resize, warn instead")
     3. Returns { base64: string, mime: string, width: number, height: number }
     4. If resize is unavailable and image exceeds limit, returns an error string instead of the base64 data

     Then patch each ingestion surface (1-5 above) to call this utility before creating the attachment. Surface 6 (
     batch.ts) just aggregates — no change needed if inputs are already validated.

     Dimension reading requires no dependency at all — PNG stores width/height at bytes 16-23 of the file, JPEG in
     the SOF0/SOF2 marker, etc. This is ~40 lines of code.

     Resizing strategy: Check if sharp is importable (try/catch dynamic import). If not, check if convert (
     ImageMagick) is on $PATH via Bun.which("convert"). If neither, fall back to dropping the image and returning a
     text error to the model: "ERROR: Image is {w}x{h} which exceeds the {max}px limit and cannot be resized (install
     sharp or imagemagick). Inform the user."

     File: src/session/message-v2.ts, in toModelMessages() around line 525-541
